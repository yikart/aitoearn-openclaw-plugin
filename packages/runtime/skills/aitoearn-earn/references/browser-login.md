# Browser 登录

## 何时读本文件

- 用户要做小红书任务，且需要站内动作或发布
- 用户问“小红书登没登录”
- 用户说要在 browser 里登录小红书
- 你准备打开小红书页面，但还不确定能不能直接继续

## 默认做法

- 先确认当前环境存在 `browser` tool
- 默认优先用 `profile="openclaw"` + `target="host"`
- 如果用户已经在自己的 Chrome 上挂好了 OpenClaw 扩展中继，再考虑 `profile="chrome"`
- 小红书链路是否可继续，只看 `browser` tool 和当前 browser 登录态；不要先查后端账号绑定状态
- 不要索要账号密码、验证码、Cookie 或登录截图
- 登录只能让用户在 browser 里手动完成

## 登录态检测优先级

小红书链路一律先判登录态，再决定是否引导登录。

推荐顺序：

1. 打开目标页或小红书首页
2. 先做一次 `snapshot`
3. 必要时再用 `act kind="evaluate"` 读取顶部导航、头像区、登录弹窗
4. 已登录就直接继续，不要再输出整段登录引导

登录态判断必须基于当前页面实际信号，不要凭“这个账号应该登录过”之类的假设继续执行。

已登录的常见信号：

- 能看到头像、个人菜单、消息入口、发布入口、创作中心入口
- 页面没有登录弹窗、手机号登录表单、验证码输入框
- 作品详情页可以直接进行评论、点赞、收藏、关注，而不是先弹登录层

未登录的常见信号：

- 页面出现“登录 / 注册”按钮
- 出现手机号、验证码、短信登录表单
- 打开详情页后先弹出强制登录弹窗或遮罩

如果判断不确定：

1. 先再做一次 `snapshot` 或短暂 `wait`
2. 再不确定时，只向用户补一句简短确认
3. 不要在未确认的情况下直接要求用户重新登录

## 引导登录

只有在明确未登录时才引导。

推荐流程：

1. 用 `browser` 打开小红书首页或目标任务页
2. 告诉用户“请在当前 browser 窗口手动完成登录，我等你登录完再继续”
3. 等用户明确说“已登录”或“继续”
4. 重新 `snapshot` 验证登录态
5. 登录成功后直接进入后续动作，不再重复登录说明

## Host Browser 约束

- 如果当前会话默认落在沙箱 browser，但站点更适合真实主机浏览器，优先切到 `target="host"`
- 如果 `target="host"` 不可用，明确说明需要允许 host browser control，或让用户在可控的 profile 中重试
- 不要在小红书这种强登录站点上默认依赖沙箱浏览器登录

## 输出要求

- 已登录：直接告诉用户“当前已登录，可以继续下一步”
- 未登录：只给最短登录引导，不展开浏览器原理
- 登录完成后：立刻收敛到具体任务动作，不停留在登录说明
